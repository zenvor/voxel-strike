# AGENTS.md

本文件适用于仓库根目录及其所有子目录。除非用户明确提出相反要求，所有开发 Agent 都必须遵守这些规则。若子目录存在更具体的 `AGENTS.md` 或 `AGENTS.override.md`，以更具体的规则为准。

## 1. 项目目标

这是 **VOXEL STRIKE // 方块前线**：一款运行在桌面浏览器中的第一人称 3D 体素射击游戏。

当前技术栈：

- TypeScript
- Three.js / WebGL 2
- Web Audio API
- Vite
- 原生 DOM 与 CSS，不使用 React、Vue 等 UI 框架

核心产品目标：

- 打开即玩，操作响应直接、清晰、稳定。
- 保持明亮、可读的白昼战场，不让环境光照妨碍战斗。
- 地形可破坏、可建造，敌人、武器、掉落和补给形成完整循环。
- 在中等配置桌面设备上保持稳定帧率。
- 生产版本能够打包为一个无需服务器、无需联网的独立 HTML 文件。

## 2. 开始任务前

在修改代码前：

1. 阅读本文件、`README.md`、`package.json`，以及与任务有关的源码。
2. 查看 `git status` 和现有改动，不要覆盖或回退用户未要求处理的修改。
3. 先确认问题的根因和受影响模块，再动手修改。
4. 默认自主作出合理的实现决定并完成任务，不要因为小细节频繁询问用户。
5. 只有在需求互相冲突、需要删除大量内容、引入在线服务、付费依赖或改变核心产品方向时，才需要先征求用户意见。

禁止提交占位实现、假按钮、未接入的 UI、仅写注释的“以后再做”方案或无法运行的伪代码。

## 3. 源码与生成文件

源码的权威来源是：

- `src/**`
- `index.html`
- `public/**`
- `vite.config.ts`
- `tsconfig.json`
- `package.json` 与 `package-lock.json`
- `scripts/**`

以下是生成产物，不得直接手工修改：

- `dist/**`
- `voxel-strike-standalone.html`

修改源码后，通过命令重新生成这些文件：

```bash
npm run build
npm run standalone
```

或一次完成：

```bash
npm run package
```

`voxel-strike-standalone.html` 不是 Vite 默认产物，而是 `scripts/make-standalone.mjs` 从 `dist/index.html` 读取 Vite 构建结果后生成的单文件版本。`npm run build` 只生成依赖 `dist/assets/` 的普通生产构建；只有继续执行 `npm run standalone`，才会把编译后的 JavaScript、CSS 和 SVG 图标内联到根目录的 `voxel-strike-standalone.html`。

`npm run package` 是推荐的完整打包命令，等价于依次执行：

```bash
npm run build
npm run standalone
```

若交付时需要带版本号的文件名，应在项目外复制或重命名生成结果；项目脚本默认输出文件名始终是 `voxel-strike-standalone.html`。不要把外部改名后的交付文件当作源码权威文件放回仓库。

保持 `vite.config.ts` 中的相对路径部署能力，确保构建产物和单文件版本能通过 `file://` 直接运行。不要让运行时依赖 CDN、远程字体、远程贴图、远程音频、统计脚本或其他网络请求。

## 4. 常用命令

首次安装依赖可使用：

```bash
npm install
```

若需要干净复现依赖、CI/发布验证，或避免无意更新 lockfile，可使用：

```bash
npm ci
```

`npm ci` 会严格按照 `package-lock.json` 复现依赖，并清理后重装 `node_modules`。`npm install` 更符合普通本地开发习惯，也用于新增或调整依赖；依赖发生变化时，应提交对应的 `package.json` 与 `package-lock.json` 变更。

本地开发：

```bash
npm run dev
```

类型检查与生产构建：

```bash
npm run build
```

生成独立 HTML：

```bash
npm run standalone
```

完整发布构建：

```bash
npm run package
```

本地预览生产构建：

```bash
npm run preview
```

不要手工编辑 `package-lock.json`。依赖发生变化时，使用 npm 命令更新锁文件。

## 5. 完成标准与验证

任何改动都必须至少运行：

```bash
npm run build
```

涉及游戏运行逻辑、输入、UI、CSS、Three.js 场景、音效、构建流程或发布内容时，还必须运行：

```bash
npm run package
```

并进行浏览器实测。根据改动范围选择最小但充分的验证，不要为了形式重复执行无关测试。

基础浏览器检查：

- 页面与主菜单正常加载。
- 控制台没有新的 error、未处理 Promise rejection 或 WebGL 警告风暴。
- 点击开始后能进入游戏并获取指针锁定。
- `Esc` 能暂停并释放鼠标，恢复后输入状态正常。
- 画布尺寸、HUD、菜单在 1280×720 与常见宽屏尺寸下不溢出。
- 单文件版本能通过 `file://` 直接打开，不访问外部资源。

玩法或输入改动的检查：

- `WASD`、跳跃、冲刺、射击、瞄准、换弹、换枪、手雷、建造和补给交互均无回归。
- 换枪、换弹、开火、暂停或失去焦点时，不会残留持续射击或卡键状态。
- 至少完成一次相关玩法路径，例如击杀敌人、拾取掉落、使用补给箱或进入下一波。
- 不得出现“弹药耗尽后永久无法继续”的死局。

依赖发生变化或准备发布时运行：

```bash
npm audit --omit=dev
```

当前仓库没有正式自动化测试套件。不要声称“测试全部通过”，除非确实运行了对应检查。若新增测试，应同时提供可执行的 npm script，并让测试覆盖真实行为而非仅覆盖实现细节。

`QA_REPORT.md` 只能记录实际执行过的验证。不能编造测试结果、帧率、浏览器兼容性或安全审计结论。

## 6. 不可破坏的产品约束

除非用户明确要求改变，下列规则优先级最高。

### 白昼与可见性

- 不得重新加入黑夜、黄昏转黑、完整昼夜循环或会长期降低可见度的天气。
- 天空、太阳和云层可以轻微变化，但战场必须始终保持明亮。
- 敌人轮廓、弹道、掉落、补给箱、准星和交互提示必须在地形背景上清晰可辨。
- 禁止用大面积黑色遮罩、过重雾效、过暗曝光、强烈逆光或长时间闪屏制造氛围。

### 灵敏度与操作

- 默认灵敏度保持低至中等水平；当前设置值为 `0.42`，实际映射由 `Game.mapSensitivity` 控制。
- 修改灵敏度范围或映射时，要同时检查设置 UI、默认值、旧设置迁移和瞄准状态。
- 不要在没有用户要求时加入鼠标加速度。
- 射击、瞄准、换枪和移动的反馈应快速，但不能通过过度镜头抖动影响瞄准。

### 战斗续航

- 敌人掉落、保底机制和地图补给箱共同保证弹药续航。
- 低生命值时必须保留医疗掉落或其他可靠恢复路径。
- 补给箱必须可到达、可识别、可重复使用，并有清楚的冷却反馈。
- 调整掉率、弹药容量、波次数量或敌人生命值时，要验证普通玩家仍能完成战局。
- 掉落物不要生成在不可达方块内部、地图外或会立即坠落消失的位置。

### 离线与隐私

- 游戏运行时不得发送遥测、分析、广告或其他网络请求。
- 不保存敏感信息。
- 新资源必须放入仓库并能进入独立 HTML；优先使用程序化模型、程序化特效和 Web Audio。

## 7. 现有操作约定

除非任务明确要求重新映射，保持以下操作：

- 移动：`W` `A` `S` `D`
- 冲刺：`Shift`
- 跳跃：`Space`
- 射击：鼠标左键
- 精确瞄准：鼠标右键
- 换弹：`R`
- 切换武器：`1`—`6` 或滚轮
- 部署金属方块：`Q`
- 使用补给箱：`E`
- 投掷电浆手雷：`G`
- 暂停与释放鼠标：`Esc`

若操作发生变化，必须同步更新：

- 游戏内提示
- 菜单说明
- HUD
- `README.md`

Pointer Lock 与 AudioContext 的启动必须继续由用户手势触发。

## 8. 代码结构与职责

- `src/main.ts`：应用入口、菜单、设置、HUD DOM、按钮事件。
- `src/style.css`：菜单、HUD、响应式布局、动画与视觉反馈。
- `src/game/game.ts`：主循环、输入编排、波次、掉落、补给箱、得分和总体状态。
- `src/game/world.ts`：体素数据、程序化地形、区块网格、碰撞查询和 DDA 射线检测。
- `src/game/player.ts`：玩家移动、碰撞、生命、护盾、体力和视角。
- `src/game/weapon.ts`：武器状态、射速、弹药、换弹、换枪、瞄准、后坐力和第一人称模型。
- `src/game/enemy.ts`：敌人模型、生命、移动、视线和战斗 AI。
- `src/game/effects.ts`：粒子、曳光、命中环、闪光等短生命周期效果。
- `src/game/environment.ts`：恒定白昼天空、太阳、雾和云层。
- `src/game/audio.ts`：Web Audio 合成音效。
- `src/game/noise.ts`：种子随机数与噪声。
- `src/game/types.ts`：武器、敌人、方块、难度和设置的共享类型与配置。

保持模块职责清晰。`game.ts` 已经较大：小改动可以就地完成，但新增大型独立系统时，应优先抽取成专门模块，而不是继续把所有逻辑堆入 `Game` 类。不要为了“重构”而改动大量无关代码。

`main.ts` 中生成的 DOM ID 与 `Game.captureUI()` 构成契约。修改任何 ID 时必须同时更新两处，并验证页面启动。

## 9. TypeScript 与编码规范

- 保持 TypeScript 编译通过，并遵守当前 `noUnusedLocals`、`noUnusedParameters` 和 `noFallthroughCasesInSwitch` 配置。
- 避免 `any`、非空断言滥用和无意义类型转换。
- 使用 `import type` 导入纯类型。
- 对外共享的数据结构集中在 `types.ts`，不要在多个模块复制相同字符串联合类型或配置。
- 热路径中避免每帧创建大量临时对象、数组、闭包和字符串。
- 对复杂数值使用有意义的常量名；不要散落无法解释的魔法数字。
- 新逻辑应处理暂停、重新开始、返回菜单、页面失焦和销毁场景。
- 不要无理由引入新的框架或状态管理库。
- 不要做大范围格式化或重命名，除非任务本身就是重构。

世界生成和需要可复现的玩法逻辑应使用现有种子随机工具。纯视觉粒子、音频噪声等不影响玩法结果的效果可使用非确定性随机数。

## 10. Three.js、资源与性能

性能是功能的一部分。修改渲染和玩法时遵守：

- 体素地形只生成暴露面，方块变化只重建受影响区块。
- 大量相似物体优先使用 `InstancedMesh`、共享 Geometry、共享 Material 或合并网格。
- 避免在每帧 `traverse` 大型场景树或更新不需要变化的对象。
- 复用 `Vector2`、`Vector3`、`Quaternion`、`Raycaster` 和临时数组，尤其是射击、AI、碰撞和粒子热路径。
- 限制粒子、弹丸、掉落、动态灯光和音频节点的最大并发数量。
- 删除对象时正确从场景移除，并释放不再共享的 Geometry、Material、Texture 和 AudioNode。
- 新增事件监听器时，要考虑重复初始化和销毁；不得产生重复监听或内存泄漏。
- 保留帧时间上限，避免切换标签页后一次性执行超大物理步长。
- 保留三档画质和“均衡”模式的自适应分辨率机制。
- HUD 与雷达应节流更新；不要每帧重复写 DOM 或重绘不变内容。
- 阴影、点光源、透明材质和后处理都必须有明确性能预算。

不要仅凭软件渲染容器中的 FPS 判断真实性能。性能改动应结合帧时间、对象数量、绘制调用、内存分配和真实浏览器观感判断。

## 11. 新增武器时

新增武器不是只添加一条配置。至少检查并完成：

- `WeaponId` 与 `WEAPONS` 配置
- 弹匣、备用弹、伤害、射速、扩散、射程、后坐力和装填时间
- 半自动或全自动触发逻辑
- 第一人称程序化模型
- 开火、换弹、空仓和换枪反馈
- 弹药补给规则
- 解锁与掉落规则
- 武器槽、名称、代号、锁定状态和按键
- 重新开始时的状态重置
- `README.md` 武器表和操作说明

新武器必须有明确定位，不能只是旧武器的数值复制。验证其不会让其他武器完全失去意义，也不会造成无限资源或无法补给。

## 12. 新增敌人或玩法实体时

新增敌人至少处理：

- 类型与基础配置
- 模型、碰撞体或命中区域
- 移动、绕障、视线与攻击节奏
- 难度倍率
- 雷达显示
- 波次生成与数量控制
- 掉落与得分
- 死亡和返回菜单时的资源清理

新增投射物、手雷、掉落、补给设施或环境实体时，必须设置生命周期、最大数量、场景移除和资源释放逻辑。

AI 不应通过墙体直接攻击玩家。涉及射线、视线或寻路的改动要同时验证可破坏地形变化后的行为。

## 13. 存档与兼容性

当前本地存储键：

- `voxel-strike-settings`
- `voxel-strike-high-score`

修改设置结构时：

- 提升 `SETTINGS_VERSION`。
- 编写向后迁移逻辑。
- 为缺失、损坏、越界或旧版本数据提供安全默认值。
- 不得无故清空最高分或用户设置。

种子相同的世界地形应尽量保持可复现。若世界生成算法发生重大变化，应在变更说明中明确记录。

## 14. 视觉、UI 与音频

- 画面风格保持清晰的科幻体素战场，不要让装饰压过玩法信息。
- HUD 在战斗中必须能快速读出生命、护盾、弹药、武器、波次和交互状态。
- 准星、命中反馈和伤害反馈应明确，但避免大面积遮挡视野。
- 菜单与 HUD 必须支持常见桌面宽屏和较低高度窗口。
- 不要依赖 hover 才能获得关键说明。
- 避免过强屏幕震动、持续高频闪烁和令人不适的相机位移。
- 音量设置必须影响所有游戏音效；创建 AudioContext 后仍需遵守浏览器自动播放限制。
- 新音效应复用或正确关闭节点，避免持续增长的音频资源。

若视觉发生明显变化，应更新预览截图。截图是发布资料，不是源码权威来源。

## 16. Git 与变更安全

- 不执行 `git reset --hard`、强制覆盖、删除 `.git` 或改写用户历史。
- 不回退与当前任务无关的已有修改。
- 不提交 `node_modules`、日志、临时文件或系统文件。

## 17. 任务完成后的报告

完成任务后，用简体中文简要说明：

1. 实现了什么。
2. 改动了哪些主要文件。
3. 实际运行了哪些构建、测试和浏览器检查。
4. 是否存在已知限制或尚未验证的内容。

不要把“代码看起来正确”写成“已经测试通过”。不要隐瞒构建失败、浏览器限制或未完成项。
